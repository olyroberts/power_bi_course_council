0. Foundations

Before Power BI makes sense, a handful of underlying tech concepts need to be solid. This tab explains them with analogies and minimal jargon. Read whatever's unfamiliar; skip what you already know.

0.1. What is a database?

A database is software for storing structured information and getting it back out quickly. The data lives in tables — like spreadsheets, but designed to handle millions of rows reliably, by many users at once, without breaking.

Analogy Think of a database like a giant, well-organised filing cabinet. Each drawer is a table (Premises, Inspections, Officers). Each file in a drawer is a row. Each piece of information on the file (name, address, date) is a column. The filing cabinet has strict rules: every Premises file has exactly these fields, no extras, no missing.

0.2. Database vs Excel — what's the difference?

Both store data in rows and columns. Differences that matter:

AspectExcelDatabase
Designed forOne person editing one fileMany people reading/writing concurrently
SizeSlows down past ~100,000 rowsHandles millions/billions
RulesFlexible — you can put anything anywhereStrict — column types enforced, relationships enforced
AccessOpen the fileConnect via a query language; user permissions per table
ReliabilityEasy to corrupt with bad editsDesigned for integrity and recovery

Uniform's data lives in a database, not in spreadsheets. This is why you can't just "open Uniform's data file" — there isn't one file. The database is a service running on a server, and you talk to it via queries.

0.3. What is a server?

A server is just a computer whose job is to provide a service to other computers. It runs 24/7, sits in a data centre or server room, and waits for requests. When Power BI asks Uniform's database for some data, it's sending a request to a server.

"On the server" usually means "running on a computer in the council's IT room, not on your laptop." This contrasts with "on the cloud" — running on someone else's servers (Microsoft, Amazon, etc.) over the internet.

0.4. On-premises vs cloud

  • On-premises ("on-prem"): the software runs on the council's own servers, inside the council's network. Uniform almost always runs on-prem because councils need control over sensitive data.
  • Cloud: the software runs on a vendor's servers, accessed over the internet. Power BI Service is cloud-based — running in Microsoft Azure data centres.

This split creates the gateway problem you'll see later — cloud Power BI Service needs a way to talk to on-prem Uniform, and that bridge is the gateway.

0.5. What is SQL?

SQL (Structured Query Language) is the standard language for talking to databases. You don't need to write SQL for this project, but you should be able to read a simple SQL query and understand what it asks for.

Pronounced "ess-cue-ell" or "sequel" — both are accepted.

Example SQL:

SELECT PremiseName, Address, RiskCategory
FROM Premises
WHERE Status = 'Active'
  AND Ward = 'Camden Town'

Read it as: "Give me the premise name, address, and risk category, from the Premises table, where status is Active and ward is Camden Town." That's the entire mental model. SQL queries are mostly just "give me these columns, from these tables, where these conditions are true."

SQL is also used to change data (INSERT, UPDATE, DELETE) — but for Power BI reading from Uniform, only the read part (SELECT) is relevant.

0.6. What is a SQL Server?

Confusingly, "SQL Server" (capital letters) is the name of Microsoft's specific database product. So "Uniform runs on SQL Server" means Uniform's data is stored in Microsoft SQL Server software. There are other database products (Oracle, PostgreSQL, MySQL) — Civica/Uniform uses Microsoft's.

So when someone says "Power BI connects to SQL Server", they mean "Power BI is talking to a Microsoft SQL Server database, querying it with SQL."

0.7. What is a query?

A query is a request to the database — "give me this data." A SQL SELECT statement is a query. Every Power BI report runs queries (often hundreds per page) to populate visuals.

0.8. What is a view?

A view is a saved query inside the database itself. Instead of writing the same SQL repeatedly, you save it under a name (like vw_PremisesActive) and then use it as if it were a table.

This matters because Power BI usually connects to views, not to raw tables. The view might already filter, join, rename — it's a pre-shaped data set built by a database administrator. If the view is wrong, every report built on it inherits the problem.

0.9. What is an API?

An API (Application Programming Interface) is a structured way for one piece of software to ask another piece of software for data or actions. Think of it like a menu of allowed requests.

Analogy An API is like a restaurant's drive-through window. The kitchen (database) doesn't let you wander in and rummage. There's a window (the API) with a menu of things you can order, and a script for how to order them. The kitchen prepares your order and hands it out. You can only get what's on the menu, formatted how they format it.

FSA pulling FHRS data from council Uniform installations happens via an API — FSA sends structured requests, Uniform's API responds with food premise data in a structured format (XML).

0.10. What is OData?

OData (Open Data Protocol) is a specific style of API designed for exposing data sets over the internet in a standardised way. Power BI can connect natively to OData feeds. Uniform offers an OData feed — meaning Power BI can pull Uniform data through OData instead of through direct SQL.

0.11. XML, JSON, CSV — formats

These are three ways to package data for transfer between systems:

  • XML — tagged format with opening and closing markers, verbose. FSA's open data feeds use XML.
  • JSON — lighter format with brackets, used widely in modern APIs.
  • CSV — comma-separated values, a plain text format like a flat spreadsheet.

Power BI can read all three. You'll work with XML when you practise on FSA data, and may encounter CSV exports from Uniform.

0.12. What is "the schema"?

The schema of a database is its structure: which tables exist, what columns each has, what data type each column holds, what relationships connect them. Think of the schema as the blueprint of the filing cabinet — the labels on each drawer and the layout of each form inside.

When someone says "the Uniform schema changed in the latest update", they mean the structure changed — maybe a column was renamed or added — which can break anything (including Power BI) that expected the old structure.

0.13. What is ETL?

ETL stands for Extract, Transform, Load — a three-step pattern for moving data between systems.

  • Extract: pull data from the source
  • Transform: clean, reshape, filter, calculate
  • Load: place into the destination

Power Query (a component of Power BI) is an ETL tool. Most ETL work in council Power BI happens inside Power Query.

0.14. Refresh and caching

Caching means storing a copy of data locally to avoid asking the source every time. Refresh means updating the cached copy from the source. Most Power BI reports cache their data and refresh on a schedule (e.g. nightly). This is why a report can show data that's 23 hours old: it hasn't refreshed yet today.

0.15. Credentials and permissions

To connect to a database or service, software needs credentials — typically a username and password, or a digital certificate. The credentials determine what data the connection is allowed to see. Different credentials may see different data.

This matters because the credentials Power BI uses to connect to Uniform may have different permissions than the credentials your husband uses to log into Uniform directly. Two people querying "the same" data can see different results because they're authenticated differently.

0.16. Versions of "the same" thing

You will encounter the same data described differently in different places — Uniform UI, Uniform mobile, FSA public site, Power BI report, an Excel export, an email summary. Each is a view of the underlying data through different filters, at different freshness, with different transformations applied. Understanding which view shows what is the entire job of this project.

1. What Is Power BI

What the name means

Power BI stands for "Power Business Intelligence."

  • Business Intelligence (BI) is a long-established industry term — it means the process of turning raw business data into insights that inform decisions. The term has been around since the 1960s.
  • "Power" is Microsoft's brand prefix for a family of analytics tools that started life as add-ins to Excel: Power Query (for loading data), Power Pivot (for modelling), Power View (for visualising). In 2015 Microsoft combined these into a standalone product called Power BI.
  • "Power" suggests power-user capability — features that previously required IT specialists, made accessible to analysts.
  • Today, Microsoft's Power family is broader: Power BI (analytics), Power Apps (build apps), Power Automate (workflows), Power Pages (websites) — collectively called the Microsoft Power Platform.

What Power BI actually does

Power BI is a Microsoft product for business intelligence — connecting to data sources, transforming the data, modelling it, and presenting it as interactive reports and dashboards. It's the dominant BI tool in UK local government because it's part of the Microsoft 365 ecosystem most councils already pay for.

It is not a single program. It is a family of products that work together:

The components you'll encounter

What gets built in Power BI

Three things, with overlapping terminology that confuses people:

  • Reports — the file (.pbix) a developer builds in Desktop. Multiple pages of visuals. Once published to Service, users open it.
  • Dashboards — in Power BI Service specifically, a single-page collection of "pinned" visuals from one or more reports. Often what executives see first.
  • Datasets (newer name: Semantic Models) — the underlying data + model + measures that one or more reports use. A single dataset can power multiple reports.
Language tip In conversation, people use "report" and "dashboard" interchangeably. When precision matters, use Microsoft's definitions above. When asking questions, just say "the Power BI thing that shows X" — everyone understands.

What Power BI is not

  • Not a database. It reads from databases (or files, or APIs). It doesn't replace where data is stored.
  • Not a data entry tool. Inspectors don't type into Power BI. They type into Uniform; Power BI reads from Uniform.
  • Not real-time by default. Most Power BI reports refresh on a schedule (e.g. every night), so the data you see is often hours old.
  • Not a single source of truth. It's a presentation of truth. If Uniform is wrong, Power BI is wrong.

2. Architecture — The Six-Layer Pipeline

This is the single most important concept in the entire course. Everything else hangs off this.

Data does not appear in a Power BI visual by magic. It travels through six layers, and any one of them can distort it. When you see a wrong number, your job is to figure out which layer broke.

One example, followed all the way through

New terms are easier to hold onto when they are attached to something concrete. So this course follows a single example the whole way through: one food safety inspection. Here is the record we will follow.

The six layers

To become a number in a report, record #4471 passes through six layers of processing. Read the diagram top to bottom. Each layer is a different colour — the colour change is a reminder that the data has been processed again, and is no longer quite what it was.

The key new ones here: source, Power Query, data model, DAX, visual, KPI card, and ward. Do not try to memorise them now — they come back, with examples, in every section.

A closer look at layer 3 — how tables "link"

Layer 3 is the one most people find abstract, so here it is drawn out. The inspection record does not store the full name "Bella Pizza" — it stores a short code, PremiseID. A separate Premises table holds the full details. A link (also called a relationship) connects the two by matching that code. The same is true for the inspector and the date.

A closer look at layer 5 — what a "visual" is

A visual is any single element on a report page that shows data. The three most common types are below. The important point: the same data shown three different ways leads the viewer to notice three different things.

How to think about each layer

1. SourceWhere is the raw data coming from?Wrong source, stale source, incomplete source view
2. Power QueryWhat was done to the data before it entered the model?Silent filters, wrong joins, type conversion errors, removed rows

The investigation method

When a user reports a wrong number, work the layers in reverse order — from layer 6 back to layer 1. The late layers are fast to check and are the most common cause; the database is the slowest thing to check and the least often at fault. Here is that method applied to a real-shaped problem.

Common mistake People investigating data problems usually start at layer 1 ("let me check the database") and waste hours. Start at layer 6 and work backwards. 70% of "wrong number" complaints turn out to be filter or measure issues, not data problems.

Power BI can read from hundreds of source types. For council work, you'll mostly see four:

Source views — the layer often overlooked

The first two rows above — SQL Server database and OData feed — are two different doors into the same Uniform data. They're easy to confuse, so it's worth being precise about how they differ.

Start with the SQL Server row. When the BI developer connects Power BI this way, they almost never connect to raw tables. They connect to SQL views — saved queries that pre-shape the data. These views might be:

  • Provided by Civica out-of-the-box (e.g. vw_PremisesActive)
  • Built by a previous BI developer or database administrator at the council
  • Inherited from a previous system and never reviewed

A view is just a stored SQL query. It might filter out closed businesses, exclude certain types, join several tables in a specific way. If the view itself is wrong or outdated, every report built on it inherits the problem.

SharePoint lists — the hidden side-tables

The last row in the table — SharePoint lists — deserves special attention, because it is the source most likely to distort a report quietly. Unlike SQL views and OData feeds, which are different routes to Uniform's own data, a SharePoint list usually holds data that is not in Uniform at all.

A SharePoint list is a simple table maintained in the browser — columns and rows, edited by hand, much like a shared Excel sheet that lives online. Council teams create them easily, without involving IT. They appear in Power BI as a side-table — an extra source "glued" onto Uniform data to enrich it. Typical examples in a food safety context:

  • Wards with their councillor names (Uniform knows the ward, not the councillor)
  • Which inspector is responsible for which premises
  • Internal team notes, target quotas, or priority flags
  • Mappings — e.g. "this Uniform business type counts as category X in our reports"

The idea is reasonable. The danger is in how these lists are maintained — they have none of the protections a real database gives:

ProblemWhy it matters for your map
No validationA person can type anything into a cell — a typo, a blank, two spellings of the same ward. Uniform would reject bad input; a list will not.
No clear ownerOften set up by an employee who has since left. Who updates it now is frequently unknown — and "nobody" is a common answer.
Goes stale silentlyNew premises, a changed inspector, a renamed ward — Uniform moves on, the list does not. Power BI still glues the outdated side-table on, with no error shown.
Invisible in the reportA viewer cannot tell which numbers came from Uniform and which from someone's hand-kept list. You can dig through Uniform and Power Query and never find the fault, because it entered from the side.

Note how this differs from the Excel / CSV row. An Excel export is a copy of data Uniform already holds — lose it, and you can re-export. A SharePoint list is usually original data that exists nowhere else — lose it or corrupt it, and the information is gone. In that sense a SharePoint list is the more fragile of the two.

Most council Power BI reports use Import mode with scheduled refresh (usually overnight). This means:

  • The numbers you see today reflect what was in Uniform at, say, 2am
  • If someone updated a record at 9am, the report won't show it until tomorrow
  • "It changed when I refreshed" usually means a scheduled refresh ran, not that the report is unstable
Question to ask the BI developer "What's the connection mode for this dataset, and what's the refresh schedule?" This single question answers a lot of "why is this number different from Uniform?" complaints — the Uniform record may have been updated after the last refresh.

The Gateway — what it is and why it matters

Power BI Service runs in Microsoft's cloud. Uniform's SQL Server runs on the council's internal network, behind a firewall. These two cannot talk directly. The bridge is the On-premises Data Gateway — a small program IT installs on a server inside the council network.

The gateway sits and waits for refresh requests from Power BI Service. When a refresh runs, the gateway queries Uniform on behalf of Power BI Service and sends results back through the encrypted channel.

Why this matters for you:

  • If the gateway is down, refreshes fail and reports show stale data with no obvious error to end users — they just see yesterday's numbers
  • Gateway credentials determine what the refresh can see in Uniform. Sometimes the gateway account has different permissions than individual users, leading to "the report shows different data than I see in Uniform"
  • Gateway is managed by IT, not the BI developer — adds a stakeholder to any troubleshooting

4. Power Query — The ETL Layer

▸ Where this fits in the data journey
The same map, every section. Data flows left to right through six layers. So far you have seen stage 1 — how record #4471 leaves the Uniform database. Right now you are learning stage 2: Power Query, where that raw record gets cleaned, filtered and reshaped before anything else touches it. This is the layer where a number is most often quietly changed, so the rest of the journey depends on getting it right. Stages 3–6 come next.

Power Query is where silent distortion happens most often. Pay close attention to this layer.

Once Power BI has pulled data from the source, before it goes into the data model, it passes through Power Query — a transformation engine.

Where Power Query actually lives

Power Query is not a separate program you install. It is a room inside Power BI Desktop. You open it, clean your data, and close it again — and you are back in the main Power BI window. Here is the layout.

Inside that editor, the BI developer can:

  • Filter rows (remove anything not matching criteria)
  • Remove columns
  • Change data types (text → number, text → date)
  • Split columns, merge columns, replace values
  • Join tables (merge queries)
  • Pivot and unpivot (turn rows into columns, or columns back into rows)
  • Group and aggregate
  • Write custom logic in the M language (Power Query's formula language)

All of these are recorded as steps in the Applied Steps panel. Each table has its own list of steps.

Pivot and unpivot — reshaping a table

Most of the operations above add, remove or change data. Pivot and unpivot are different — they don't change the data, they change its shape: which way the table is laid out. This sounds harmless, but a reshape done carelessly is a classic source of doubled or lost numbers, so it is worth seeing clearly.

Why this is the most dangerous layer

Power Query transformations are invisible to end users. A report consumer looking at a visual has no way to know that:

  • A "Filtered Rows" step removed all records where Status = 'Closed' — so "active premises" looks correct, but anyone clicking through expecting to see all premises is wrong
  • A "Changed Type" step converted a column to number but failed silently on rows where the value was non-numeric — those rows now have null in that column
  • A "Merged Queries" step is doing an inner join, dropping rows from the left table that don't have a match in the right
  • A "Replaced Values" step is changing every "FHRS 0" to "Awaiting Inspection" because someone thought that was clearer

The applied steps panel

The single most useful thing in Power Query Editor is the Applied Steps panel on the right-hand side. It is a numbered list of every transformation, in the order it happened. You can click any step to see the data exactly as it looked at that moment — before and after that change.

Think of it like the "undo history" in Word, but permanent and visible. Below is what the panel looks like for the Premises table — the same table that holds the Bella Pizza record.

The three red steps are the ones most likely to make a Power BI number disagree with what you would see in Uniform. Each is a legitimate thing a developer might do — but each is also a place where data quietly changes:

  • Filtered Rows (step 5) — closed premises vanish. "Total premises" in the report is really "total active premises", but nothing on screen says so.
  • Replaced Value (step 8) — wording is rewritten. Someone searching the report for "Cat A" finds nothing, because it now says "High Risk".
  • Merged Queries (step 9) — joining tables can drop rows that have no match. Covered in detail in section 5.
How this helps your investigationWhen a number looks wrong, asking the BI developer to "click through the Applied Steps for that table" is one of the most direct things you can do. You are looking for exactly these kinds of steps — and you can see the row count change as you click down the list.

The M language

Power Query records steps as M code. You can read it via the "Advanced Editor" button. You don't need to write M, but you should be able to recognise common patterns:

let
    Source = Sql.Database("server", "UniformDB"),
    vwPremises = Source{[Schema="dbo", Item="vw_Premises"]}[Data],
    #"Filtered Rows" = Table.SelectRows(vwPremises, each [Status] = "Active"),
    #"Removed Columns" = Table.RemoveColumns(#"Filtered Rows", {"InternalNotes", "AuditFlag"})
in
    #"Removed Columns"

Reading that, you can see: connected to UniformDB SQL Server, selected the vw_Premises view, kept only rows where Status is "Active", dropped two columns. If you suspect a problem in Power Query, ask the developer to show you the M code for the table in question.

Query folding — performance gotcha you should know about

Query folding means Power Query converts your steps into a SQL query and pushes them to the source database, instead of pulling all data and transforming locally. This is faster but only works for some sources and some operations.

First — Power Query and a SQL query are not the same thing

That one sentence packs in two things that are easy to confuse, so it is worth slowing down. Power Query and a SQL query are two different languages, running in two different places:

Power QueryA SQL query
What it isAn editor inside Power BI. You work by clicking — "remove this column", "keep only Active rows". Each click is recorded as a step.A line of text the database understands, e.g. SELECT Name FROM Premises WHERE Status='Active'.

Here is the problem this creates: the Uniform database cannot read M at all. It only understands SQL. So Power BI has a choice about how to get your steps done.

What folding actually does — it translates

When folding works, Power Query looks at your M steps and thinks: "all of this can be described as one SQL query." It translates them into SQL and sends that to the database. Uniform does the filtering itself and returns only the small finished result. The word "converts" in the sentence above means exactly this — a translation from one language to the other.

When folding breaks, one of your steps has no SQL equivalent — often custom M logic, or a step in an order SQL cannot express. Power Query cannot translate it, so instead it pulls the whole table across to your computer and does the work there. Everything after the un-foldable step is also processed locally.

The translator analogy You ask an assistant, in English, to "find the active cafés in this ward." But the archive clerk only speaks SQL. A good assistant translates your request and the clerk fetches just the right folder — you get back a thin file. An assistant who cannot translate instead wheels out the entire archive and sorts through it in front of you. Query folding is the assistant who can translate; broken folding is the whole archive on a trolley.

▸ Show diagram — folding on vs folding broken

Translation, not magic. Folding does not change what your steps do — it changes where they run. Working: the database filters and returns little. Broken: everything crosses to your machine first. The result is the same data; the cost is speed and refresh reliability.

Why you should care: if folding breaks, refreshes get slow and can time out, leading to partial data refreshes. This sometimes causes "today the report has 4000 rows, yesterday it had 4200" — not a data quality issue per se, but a refresh integrity issue.

Question to ask the BI developer "Do the queries for this dataset fold all the way to the source?" In Power Query Editor, a developer can right-click the last step and check whether "View Native Query" is available — if it is greyed out, folding has broken at that step. A folded query is also one you can ask to see: the native SQL it generates is often the clearest description of what the report is really pulling from Uniform.

5. The Data Model

▸ Where this fits in the data journey
The same map, every section. Data flows left to right through six layers. You have already seen how it leaves Uniform (stage 1) and gets cleaned in Power Query (stage 2). Right now you are learning stage 3 — how those cleaned tables get linked into a model. Stages 4–6 come next. Any of these layers can distort a number, which is why the map stays with you the whole way.

Once data has been pulled and transformed, it enters the data model — Power BI's internal representation of multiple tables linked together. You see this in Model View (third icon on the left in Desktop).

The star schema

A well-built Power BI model follows a star schema:

  • One central fact table — contains events or measurements (e.g. Inspections, with columns: InspectionID, PremiseID, Date, Officer, Outcome)
  • Several dimension tables around it — describe entities (Premises, Officers, Dates, Outcomes)
  • Each dimension links to the fact via a foreign key
Why this shape, and why it matters to youThe fact table holds the events — thin rows of mostly ID codes and numbers. Each dimension holds the descriptions those codes point to. Keeping them separate is what lets a report slice inspections by ward, by risk category, by officer, by month — all at once. When a number misbehaves only under certain slicers, the cause is almost always one of these dimension links, not the fact table itself.

For food safety reporting, fact tables you might see:

  • Inspections — one row per inspection event
  • Service Requests — one row per complaint
  • Notices — one row per enforcement action issued
  • Samples — one row per food sample taken

And dimension tables:

  • Premises — one row per food business
  • Officers / Staff — one row per inspector
  • Dates — one row per day, with year, month, quarter, fiscal year columns (called a "date dimension" — essential for time analysis)
  • Outcome Codes — lookup for what each code means
  • Risk Categories — Cat A, B, C, D, E with descriptions

Relationships

Tables are linked by relationships. Each relationship has properties that determine behaviour:

Cardinality

Cardinality is just a plain question: when I take one row from table A, how many matching rows does it have in table B? Take Premises and Inspections. Pick one premise — say "Rose Café." It was inspected in January, March and August. So one premise → many inspections. Now go the other way: pick one inspection — which premise does it belong to? Exactly one. An inspection cannot belong to two cafés at once.

  • One-to-many (1:*): the normal case, and the answer above. One premise has many inspections; the "one" side is the dimension, the "many" side is the fact. Real life is nearly always like this: one patient → many visits, one vehicle → many fines.
  • Many-to-many (*:*): both sides are "many" at once — and in Power BI this is usually a warning, not a real relationship. A clean example where it is genuine: students and courses (one student takes many courses; one course has many students). But in a council model it almost always means duplicates have crept into a table that should be a unique lookup. If Premises accidentally arrives with PremiseID 101 listed twice, then 101 is no longer unique on the dimension side either — so Power BI gives up and calls the link *:* . The consequence is double-counting: each inspection matches both copies of 101 and totals inflate. This is exactly why the model checklist asks whether dimension tables are unique on their key.
  • One-to-one (1:1): rare and usually wrong — these tables should probably be combined.

Filter direction

Filter direction is simply which way a click travels. When you pick Officer "Ahmed" in a slicer, Power BI has to decide which inspections to show. The arrow on the relationship says the filter flows from the dimension (Officers) down to the fact (Inspections) — so picking an officer narrows the inspections to his. That is single direction, and it is correct almost every time.

  • Single direction (the default and usually correct): filtering the dimension filters the fact, but not the reverse. Pick an officer, see his inspections; pick a risk category, see those inspections. The filter only flows downhill.
  • Both / bidirectional: filters flow both ways, so a choice in one dimension can reach back through the fact and affect another dimension. It sounds convenient but it is a common cause of double-counting and ambiguous results — the filter can travel in a loop and Power BI is no longer sure which number to show. Many BI developers learn to avoid it. If you see it in a model, the question to ask is "why is this on?"

▸ Show diagram — single vs both direction

The arrow is the rule. By default the filter only travels from the describing table down to the events table. Turn it both ways and a choice can loop back through the fact into another dimension — which is where double-counting and "the number depends on which slicer I touch first" come from.

Active vs inactive

This comes up whenever there are two possible relationships between the same two tables — most often because the fact table holds more than one date. Imagine each inspection carries two dates: the date it was scheduled and the date it was actually completed. You also have a Dates calendar table. You could sensibly link Inspections to Dates by either date — so there are two candidate relationships to one table.

Power BI lets you keep both, but only one can be active at a time (drawn as a solid line). The other lies dormant — inactive (a dotted line). Every ordinary report counts through the active one (say, CompletedDate). When you specifically need the other one — "how many inspections were scheduled in March?" — you temporarily wake the dormant relationship inside a single DAX measure using USERELATIONSHIP.

▸ Show diagram — active vs inactive relationship

Why this trips people up. You open "inspections by month" and it looks fine — but it is counting by CompletedDate. Someone else's report counts the same March by ScheduledDate (via USERELATIONSHIP) and shows a different figure. Both are right; they just measure different dates. The dotted line is easy to miss, so the gap looks like "the report is broken" when nothing is.

Reading a model diagram

In Model View, you see boxes (tables) connected by lines (relationships). Each line has:

  • An arrow indicating filter direction
  • A "1" or "*" at each end showing cardinality
  • A solid line for active, dotted line for inactive
What to check in any model
  1. Is there a star schema, or is it a "snowflake" (dimensions linking to other dimensions) or, worse, a "spaghetti diagram" with relationships everywhere?
  2. Are there any many-to-many relationships? If yes, why?
  3. Are there any bidirectional filters? If yes, why?
  4. Is there a proper Date dimension table? "Proper" means a dedicated calendar table — one row per day, a continuous range with no gaps (including days when nothing happened), and pre-built columns for year, month, quarter, week number and fiscal year. Not proper is when there is no such table and people filter straight off a date column in the fact table. The difference matters: without a real calendar, months with no inspections silently drop out of the report, months sort alphabetically ("April, August, December…") instead of chronologically, and time-intelligence calculations (year-to-date, year-over-year) break. So this question really asks: is there a dedicated calendar, or is the date being pulled ad hoc from the facts?
  5. Are dimension tables actually unique on their key column? (If Premises has two rows with PremiseID = 123, all reports counting through this dimension will double-count.)

Orphaned records

If a fact row has a key that doesn't match any dimension row, it's "orphaned". Example: an inspection with PremiseID = 9999, but no premise 9999 exists in the Premises table.

In Power BI, orphaned rows still appear in totals but show up as "(blank)" when sliced by that dimension. This is a classic source of "the total doesn't add up to the sum of categories."

Classic council data issue Premises get merged or deleted in Uniform. Old inspection records still reference the deleted PremiseID. In Power BI, these inspections appear in totals but vanish when you filter by ward, premise type, etc. People see different numbers depending on which slicer is on. The root cause is orphaned facts, but it looks like "the report is broken."

Calculated columns vs measures

Two ways to add computed values:

  • Calculated columns: computed row by row when the model is refreshed. Stored in the table. Visible as columns. Good for: categorisations, lookups, sorting helpers. Examples on inspection data: a RiskBand column that reads "High" if the rating is 4–5 and "Low" otherwise; a DaysSinceInspection column = today minus the inspection date; a FinancialYear column derived from the date so you can sort and group by it. Each of these answers a per-row question — one value per inspection — which is exactly what a column is for.
  • Measures: computed at query time based on the current filter context. Not stored. Good for: aggregations (sums, counts, percentages). Examples: Total Inspections = COUNTROWS(Inspections); Avg Rating = AVERAGE(Inspections[Rating]); % High Risk = DIVIDE([High Risk Count], [Total Inspections]). The key difference: a measure gives a different answer depending on what is filtered — slice by ward and "Total Inspections" recalculates for that ward automatically. Most reporting logic should be measures, not columns.

The quickest test: if the answer is one value per row, use a column; if the answer is one number for a whole group that should change as filters change, use a measure. "What risk band is this inspection?" is per row → column. "How many inspections are there?" depends on the current slicers → measure.

Confusing the two leads to wrong results. Take the Total Inspections example above: done right, it is a measure and recalculates per slice. The classic mistake is building it instead as a calculated column that counts all inspections — every row gets the same grand total, frozen at refresh time, ignoring filter context. People then drop that column into visuals and wonder why filtering by ward, month or officer does nothing. Same name, same intent, wrong tool — and the report looks broken.

6. DAX & Measures

DAX (Data Analysis Expressions) is Power BI's formula language. It looks superficially like Excel formulas — SUM, COUNT, IF — but the way it evaluates is fundamentally different and is where many "wrong number" mysteries originate.

You don't need to write DAX. You need to read it.

When investigating a number, you'll ask: "What's the DAX for this measure?" The BI developer will show you something like:

Active Premises Count = 
CALCULATE(
    DISTINCTCOUNT(Premises[PremiseID]),
    Premises[Status] = "Active",
    Premises[InScope] = TRUE
)

Even without writing DAX, you should be able to read this and say: "this counts distinct premise IDs where status is Active and InScope is True."

The most important DAX functions to recognise

FunctionWhat it doesCommon error
SUM(column)Adds up numbers in a columnSumming IDs (meaningless but doesn't error)
COUNT(column)Counts non-blank values in a columnCounting all rows when you wanted distinct values
COUNTROWS(table)Counts rows in a tableCounts duplicates if they exist
DISTINCTCOUNT(column)Counts distinct values, ignoring duplicates and blanksOften the correct alternative to COUNT
CALCULATE(expression, filter1, filter2)Computes the expression with modified filter contextThe single most powerful and confusing function in DAX
FILTER(table, condition)Returns a filtered tableUsed inside CALCULATE for complex filters
DIVIDE(num, denom)Safe division — returns blank instead of error on divide-by-zeroForgetting to use it; getting #DIV/0 errors
RELATED(column)Pulls a value from a related table (the "one" side)Only works in calculated columns, not measures usually
SUMX(table, expression)Iterates a table, evaluates expression per row, sumsPerformance trap when overused
SAMEPERIODLASTYEAR / DATEADDTime intelligence functionsRequire a proper Date dimension to work correctly

COUNT vs DISTINCTCOUNT — the single most common bug

Two measures that sound the same but give different answers:

Inspections Done = COUNT(Inspections[InspectionID])
Premises Inspected = DISTINCTCOUNT(Inspections[PremiseID])

If 100 inspections were conducted at 80 unique premises (some premises inspected twice), then:

  • Inspections Done = 100
  • Premises Inspected = 80

Both might be labelled "Inspections" on a dashboard depending on who built it. Always ask which one is being shown when a number doesn't match expectations.

Filter context — the real source of confusion

This is the hardest DAX concept and where most "the formula looks right but the number is wrong" issues come from.

Every visual in Power BI applies a filter context to the measure. If a chart breaks inspections down by ward, the measure for the "Camden" bar runs with an automatic filter Ward = "Camden" applied. The measure has no idea this filter exists — it just runs against the filtered subset.

CALCULATE modifies filter context. It can add, remove, or override filters that came from the visual. This is incredibly powerful and incredibly easy to misuse.

// This measure ignores any filter on Officer and counts all premises:
All Premises = CALCULATE(DISTINCTCOUNT(Premises[PremiseID]), ALL(Officers))

// This measure adds a hard filter for active only, regardless of what user filtered:
Active Premises = CALCULATE(DISTINCTCOUNT(Premises[PremiseID]), Premises[Status] = "Active")

If a measure uses ALL(), REMOVEFILTERS(), or hardcoded filters in CALCULATE, it can give different results than users expect when they apply slicers. The slicer is applied, but the measure overrides part of it.

Diagnostic question "Does this measure use CALCULATE with any filter arguments? If so, which filters does it override or ignore?" If yes, the measure has built-in behaviour that may not match what the user thinks they're filtering for.

Blank handling

DAX treats BLANK as a special value:

  • COUNT ignores blanks; COUNTROWS includes rows with all blanks
  • SUM treats blanks as zero, but AVERAGE ignores blanks (changes the denominator)
  • Comparison with blank gives unexpected results: BLANK() = 0 is TRUE, but BLANK() < 1 is also TRUE in some contexts

This is the cause of many off-by-N discrepancies, especially in averages.

Time intelligence and the Date dimension

Functions like SAMEPERIODLASTYEAR, YTD, MTD, DATESBETWEEN require a proper Date dimension table marked as a date table. If the model doesn't have one — or has a "broken" one with gaps or non-contiguous dates — time intelligence silently gives wrong answers.

For council reporting, fiscal year (April–March) is standard. The Date dimension must include a fiscal year column. If the developer didn't set this up, YTD calculations may be using calendar year instead — a common silent error in regulatory reporting.

Once data has reached the visual, the developer chooses how to display it. The same number, shown differently, tells different stories — sometimes deliberately, sometimes accidentally misleading.

The totals problem

Power BI tables and matrices show totals at the bottom by default. The total is not the sum of the visible rows. It's the measure evaluated in the total context — which can produce a different number.

Example: a table of premises by ward with column "Premises Inspected" = DISTINCTCOUNT(Inspections[PremiseID]).

  • Camden row: 45 premises inspected (45 distinct PremiseIDs in Camden's inspections)
  • Islington row: 38 premises inspected
  • Hackney row: 52 premises inspected
  • ... etc
  • Total row: 487 premises inspected

The total is not 45+38+52+...! It's the distinct count across all wards combined. If a premise is somehow attributed to two wards (data issue), it would count once in the total but once in each ward — sum of rows would exceed total. If a premise has no ward (orphan), it counts in the total but appears in no row — sum of rows would be less than total.

This is mathematically correct DAX behaviour but routinely confuses users who expect "totals to add up."

Diagnostic value When the visible rows don't sum to the total, that's information about the data, not a bug. It usually means: (a) the measure uses distinct count, and the categories overlap, or (b) some rows have blank category and appear in total but no category row, or (c) bidirectional filtering is causing weird aggregations.

Top-N and "Other"

Visuals can be set to show "Top 10" with everything else either hidden or grouped as "Other". If a user doesn't notice the top-N setting, they think they're seeing the full picture.

In food safety dashboards, this often hides: small ward with one problematic premise, less-common premise types, infrequent outcome codes.

Truncated axes

By default, Power BI auto-scales axes. A chart showing inspections from 240 to 260 with the Y-axis starting at 235 makes a 4% variation look dramatic. The opposite is also common: an axis starting at zero hides meaningful variation.

Aggregation set on the visual

For columns dropped directly into visuals (not via measures), Power BI auto-picks an aggregation. A numeric column defaults to Sum. So if someone drops InspectionID into a visual, Power BI sums it up — producing a nonsense number that looks plausible because it's big.

Always check: is each value in the visual coming from a properly-named measure, or from a raw column with default aggregation?

Drill-through and drill-down

Two features that confuse end users:

  • Drill-down: clicking a hierarchical visual to go deeper (e.g. Year → Quarter → Month → Day). The visual itself changes.
  • Drill-through: right-clicking a value to navigate to a different report page with filters pre-set to that value.

If a user is "drilled in" without realising, they see a filtered subset and may report it as a wrong total.

Tooltips

Hovering over a visual shows a tooltip with additional values. Tooltips can include their own measures, separate from what's plotted. Sometimes the tooltip and the visual show different counts — usually because they're using different measures.

Concept check

8. Filters & Slicers

Filters are the most common cause of "wrong number" complaints — usually because there's a filter the user didn't notice.

The hierarchy of filters

Power BI applies filters in a specific order, with later filters overriding earlier ones where they conflict:

  1. Visual-level filters — applied only to one visual on a page
  2. Page-level filters — applied to every visual on one page
  3. Report-level filters — applied to every page in the report
  4. Slicers — visual widgets users interact with
  5. Cross-filtering — clicking a value in one visual filters others on the same page
  6. Drill-through filters — when a user drilled through from elsewhere
  7. Row-level security (RLS) — applied automatically based on user identity

End users see slicers and clearly see cross-filtering, but visual-, page-, and report-level filters are hidden in the Filter pane which they may not have open or may not notice.

The "saved state" trap

When users click a slicer in Power BI Service, their selection is saved for next time they open the report. Two users opening the same report can see different numbers because they left different slicers active. This is especially confusing when one says "I see 200" and another says "I see 350" — both are technically correct for their respective filter contexts.

Diagnostic step zero Before investigating any "wrong number" complaint, ask: "What slicers do you have selected, and have you opened the Filter pane to check what's filtered?" Have them screenshot the entire report page including the Filter pane. Many investigations end here.

Cross-filtering between visuals

Clicking a bar in a bar chart filters all other visuals on the page to that bar's value. This is intended behaviour but easy to forget. Users see "wrong" numbers in other visuals when they've left a value selected somewhere.

Sync slicers

A slicer can be set to sync across pages — filtering page 1 silently filters page 2 and 3. Users navigating to page 3 see filtered data and have no visual cue why.

Row-level security (RLS)

RLS is a feature where Power BI filters data based on who's looking. A team leader might see their own team's records; their manager sees the whole borough. Same report, different data per user.

When two people compare numbers on the "same" report and disagree — RLS is one possible cause. They're literally not looking at the same data.

For council deployments: RLS is sometimes used so different teams only see their data. It's worth asking the BI developer "is there any row-level security on this dataset?"

Filter audit checklist

When a number looks wrong, before anything else:

  1. Open the Filter pane on the right (View → Filters if hidden)
  2. Note every active filter at visual, page, and report level
  3. Note all slicer selections on the page
  4. Note if any visual on the page has a value selected (cross-filtering)
  5. Right-click empty space → check for drill-through context
  6. Note who you're logged in as (RLS context)
  7. Click "Reset to default" if available, see if number changes

9. Refresh & The Service

Most council reports use Import mode and refresh on a schedule. Understanding the refresh cycle answers a lot of "why is this different from Uniform?" questions.

The lifecycle of a report

  1. Developer builds the report in Power BI Desktop, connecting to Uniform via SQL or OData
  2. Developer publishes to Power BI Service (cloud) — the dataset and report are uploaded
  3. Developer configures scheduled refresh in Service, pointing to the gateway for on-premises sources
  4. At scheduled time (e.g. 2am daily), Service triggers refresh; gateway runs queries; data flows back to Service; dataset is updated
  5. Users open the report in Service or Mobile and see data as of last successful refresh

The Service interface

app.powerbi.com is where users actually consume content. Key areas:

  • Workspaces — containers of reports, dashboards, datasets. A council might have "EH Reporting", "Licensing", "Corporate KPIs" etc.
  • Apps — published collections that users subscribe to. Cleaner consumer experience than browsing workspaces.
  • My Workspace — personal area; nothing important should live here long-term
  • Datasets / Semantic Models — the data + model files
  • Reports — the report files (linked to a dataset)
  • Dashboards — pinned-visual collections

Licensing model — relevant to know about

Power BI has confusing licensing. You don't need to master it, but you should know:

  • Power BI Free — can use Desktop, can publish to personal workspace; cannot share via Service
  • Power BI Pro — per-user paid license; can publish, share, collaborate in workspaces
  • Power BI Premium Per User (PPU) — Pro plus more features
  • Power BI Premium — capacity-based; large organisations buy a "premium capacity" and users get viewer access without Pro licenses

Most councils have Premium because it's cost-effective at scale. Some have Pro-only and that means every consumer needs a license.

Why this matters for you

If you ever get access to the council's Power BI workspace:

  1. Find the relevant dataset
  2. Refresh history tells you when data is actually as of
  3. Dataset settings tells you the data source and gateway used
  4. If a report shows "wrong" data, refresh history is the first thing to check

10. Where Data Goes Wrong — A Catalogue

This is the most useful tab for your project. A catalogue of distortion patterns you should expect to find, organised by layer.

At the source (Uniform)

  • Duplicate premise records — same physical business entered twice under variant spellings. Power BI counts them as two.
  • Inconsistent classification codes — different officers use different codes for the same situation. Filtering by code misses records.
  • Free-text fields with structured info — type of food in a "notes" field instead of a coded field. Cannot be reliably aggregated.
  • Wrong outcome codes — inspector clicks the wrong option in a dropdown; data looks fine but is mis-categorised.
  • Closed premises still flagged active — business closed two years ago, status never updated. Counted as active.
  • Missing UPRNs — premises without unique property references can't be matched to address/ward data.
  • Inspection date inconsistencies — date of visit vs date entered vs date of report all different; reports may use the wrong one.

At the source view / extract

  • View filters out records you'd expect — e.g. vw_PremisesActive filters by a status that's defined differently than the user expects.
  • View joins introduce duplicates — joining premises to addresses might multiply rows if a premise has multiple addresses.
  • View hasn't been updated — Uniform schema changed (e.g. new outcome codes added) but view wasn't extended.
  • View is built on stale base tables — if Uniform uses staging tables that don't refresh in real time, view shows stale data.

In Power Query

  • Restrictive "Filtered Rows" step — removes records that should be there.
  • "Removed Duplicates" on non-unique key — silently drops distinct records that share a value.
  • Type conversion errors — text-to-number conversion fails on some rows, leaving nulls.
  • Inner join instead of left join — fact rows without matching dimension drop out entirely.
  • "Replaced Value" hardcoded — values changed from source representation to something else, causing mismatch with Uniform.
  • Refresh timeout produces partial load — table appears refreshed but is incomplete.

In the data model

  • Wrong cardinality — many-to-many where one-to-many was intended; produces inflated counts.
  • Bidirectional filtering causing double-count — filters propagate where they shouldn't.
  • Inactive relationship being used unexpectedly — DAX function activates a relationship the user didn't realise existed.
  • Orphaned facts — fact rows with no matching dimension; appear in totals but not slices.
  • Duplicate keys in dimension — Premises table has two rows for the same PremiseID; everything joined to it doubles.
  • Missing Date dimension — time intelligence functions silently wrong.
  • Date dimension with gaps — missing days break YTD calculations.

In DAX measures

  • COUNT used where DISTINCTCOUNT was intended — most common error.
  • SUM on ID columns — meaningless aggregation; produces big numbers that look real.
  • CALCULATE with hardcoded filters — overrides user slicers in unexpected ways.
  • Filter context misunderstanding — measure gives different numbers in different visuals because of context.
  • Blank handling — averages and percentages distorted by how blanks are treated.
  • Wrong relationship used in CALCULATE — USERELATIONSHIP activates an inactive relationship that brings unexpected results.
  • Time intelligence against wrong date column — using inspection date when fiscal calculations need notification date.

In visualisations

  • Default aggregation on dropped column — Power BI sums an ID column instead of counting rows.
  • Top-N hiding the long tail — visual shows top 10, user thinks it's the full picture.
  • Truncated axes exaggerating change — small variations look dramatic.
  • Totals not matching row sums — distinct counts cause this; technically correct but confusing.
  • Tooltips showing different measure than chart — different numbers when hovering vs visually.
  • Conditional formatting using wrong measure — colours don't match actual values shown.

In filters and interaction

  • Slicer left selected from last session — user sees filtered subset thinking it's the whole.
  • Hidden visual/page/report filter — applied filter user can't see.
  • Cross-filtering from accidental click — visual filtered by another visual's selection.
  • RLS filtering by user identity — different users see different numbers on same report.
  • Drill-through context lingering — user is on a filtered page without realising.
  • Sync slicers — filter applied on one page silently applied on others.

In refresh and Service

  • Refresh failed silently — data is older than users think.
  • Partial refresh — some tables current, others not — internally inconsistent.
  • Gateway down — refresh failure pattern.
  • Different report consuming different dataset — two "Inspections" reports built off two datasets refreshing on different schedules.
  • Cache issues — Service occasionally serves cached data briefly after a refresh.

Above all layers — governance

  • Multiple "official" reports on the same metric — different teams built their own, each tells a different story
  • No documentation of measure definitions — "Inspections" means different things in different reports
  • No data dictionary — fields and codes mean different things to different people
  • No version control — current version differs from what was shown last quarter; no audit trail
  • No ownership — when something breaks, no one knows whose job it is to fix
The hardest cases The hardest data quality issues to diagnose are compound ones: a Power Query filter combined with a measure quirk combined with a slicer left active. Each alone produces plausible numbers, but together they produce numbers that almost make sense and waste hours of investigation. Always investigate one layer at a time, isolating it.

11. Council Context — Power BI for Food Safety & Regulatory Data

Generic Power BI knowledge is necessary but not sufficient. This tab covers what's specific to your project's domain: reporting on food safety, health & safety, and licensing data from a London Borough Council.

What metrics matter to FSA / regulators

The Food Standards Agency (FSA) sets expectations for local authority food safety services. Reporting typically covers:

  • Inspection coverage — % of due inspections completed in period (the "behind schedule" metric your husband is dealing with)
  • FHRS rating distribution — how many premises at each rating 0–5
  • Risk category profile — distribution of premises across Cat A (highest risk) to Cat E or "unrated"
  • Enforcement activity — notices issued, prosecutions, prohibitions
  • Service request volumes — complaints received, response times
  • Sampling — samples taken, results, follow-up
  • Premises register accuracy — newly registered, closed, awaiting inspection

The two annual returns you'll hear about

ReturnTo whomWhat it asks
LAEMS (Local Authority Enforcement Monitoring System)Food Standards Agency, annualComprehensive numbers on food safety activity, staffing, premises register status
LAE 1 returnHSE, annualHealth & Safety enforcement activity by local authority

These returns must be reconciled with reality. They're the most visible external use of Uniform / Power BI data, and discrepancies between returns and operational reports are politically sensitive.

FSA's FHRS data feed

Separately from LAEMS, the FSA continuously pulls FHRS rating data from councils for the public ratings.food.gov.uk website. This pull is direct from each council's system (Uniform in your case), via an automated feed Civica configures.

This is why FSA's public data may differ from council's Power BI:

  1. Different filters — FSA only takes "in scope, rated, published" records; Power BI may include broader set
  2. Different schedules — FSA pulls on its own cadence; Power BI refreshes on council's
  3. Different reference points — FSA gets "current published rating"; Power BI may show "most recent inspection outcome" which is different in the case of pending re-rate requests

Reconciling council Power BI against FSA's public feed is one of the highest-value early exercises for your project.

Categories of food premises

Premises are classified by risk for inspection frequency purposes. The categorisation system (FSA Code of Practice) is broadly:

  • Category A — highest risk; inspect every 6 months
  • Category B — every 12 months
  • Category C — every 18 months
  • Category D — every 2 years
  • Category E — alternative enforcement strategies; e.g. self-assessment
  • "Outside the programme" — out of scope for routine inspection
  • "Awaiting inspection" — not yet rated

For Power BI reporting, the "behind schedule" metric usually compares due date (based on last inspection + category interval) against today. Any premise where due date is in the past is overdue.

Common Power BI report patterns in EH

  • Inspection performance dashboard — KPIs on done vs due, by category, by officer
  • Premises register summary — total premises, new registrations, closures, by ward/type
  • FHRS rating distribution — histograms and trends
  • Service request tracker — open vs closed, by type, response time
  • Officer workload — inspections per officer, planned vs actual
  • Hot spots / geographic — map of low-rating premises by ward

Likely current-state problems specific to council

Based on what you've described and on common patterns in similar councils:

  1. Tablet entry inconsistency — officers in field entering data through mobile interface with reduced validation. Field-entered records often have more errors, especially in coded fields.
  2. Multiple admin staff entering differently — no SOP means each interprets dropdown choices and free-text fields differently.
  3. Behind on inspections → backlog data noise — when team is rushing to catch up, data quality drops further (corner-cutting on data entry).
  4. Power BI built incrementally without governance — different reports built at different times with different measure definitions; no one knows which is "right."
  5. FSA returns prepared by manual extract + spreadsheet — even though Power BI exists, the actual returns may be calculated outside it. Two sets of numbers, both "official."

Early-investigation questions to ask

  1. Who built the existing Power BI reports? When? Are they documented?
  2. How are FSA returns currently prepared — from Power BI, from Uniform directly, or via Excel?
  3. What's the difference between numbers in current Power BI and the most recent LAEMS submission?
  4. How does council's Power BI count of FHRS-rated premises compare to FSA's public feed for the same authority?
  5. Where does the "inspections behind schedule" calculation come from — is it a Power BI measure or a manually-curated spreadsheet?
  6. Who owns each Power BI workspace? When was each dataset last refreshed?

12. Hands-On Practice

The best way to ground these concepts is to build something small. Below is a recommended practice path using publicly available FSA data — no council data needed, no risk to confidentiality, but you'll work with the exact same kind of data your council deals with.

Practice setup (one-time)

  1. Download Power BI Desktop (free, Windows only) from powerbi.microsoft.com/desktop
  2. Install on a Windows machine. If you're on Mac, you'll need a Windows VM or you can use Microsoft Learn's sandboxed labs.
  3. Sign up for a free Microsoft account if you don't have one (a personal one is fine for learning)

Practice exercise 1 — Load and explore FSA data

FSA publishes Food Hygiene Rating data as XML feeds by authority. Find a London borough's feed at ratings.food.gov.uk/open-data.

  1. Pick a London borough (e.g. Camden, Westminster — anywhere other than the one your husband works at)
  2. Download the XML file
  3. In Power BI Desktop: Get Data → XML → load the file
  4. Observe the table structure — Power Query Editor will open. Note the Applied Steps that Power BI auto-generates
  5. Don't transform anything; just load
  6. In Report View, drop the rating value into a visual; create a bar chart of count of premises by rating

Goal: experience the full flow from source to visual. Notice everything Power BI did automatically.

Practice exercise 2 — Find the same authority's data discrepancy

Reproduce a council-style problem:

  1. In Power Query Editor, add a Filtered Rows step keeping only ratings 4 and 5
  2. Note how the original count drops
  3. Return to Report View — the visual updates, the lower ratings disappear
  4. Now imagine if a user asked "how many premises in this borough?" expecting the full count. The visual would mislead.
  5. Remove the filter; note how it returns

Goal: feel how Power Query transformations silently distort downstream visuals.

Practice exercise 3 — Build a star schema

  1. Download two boroughs' data
  2. Create a separate Authorities dimension table (manually in Power BI: Enter Data, two rows with AuthorityID and AuthorityName)
  3. Add an AuthorityID column to each premises table
  4. Append (combine) the two premises tables into one
  5. Create a relationship from Premises to Authorities
  6. Build a visual showing premises count by authority by rating

Goal: understand fact-dimension relationships in practice.

Practice exercise 4 — DAX measures

Using the combined data:

  1. Create a measure: Total Premises = COUNTROWS(Premises)
  2. Create a measure: Average Rating = AVERAGE(Premises[RatingValue]) (you may need to convert rating to number first)
  3. Create a measure: Top-Rated Premises = CALCULATE(COUNTROWS(Premises), Premises[RatingValue] >= 4)
  4. Create a measure: % Top-Rated = DIVIDE([Top-Rated Premises], [Total Premises])
  5. Build a card visual for each; build a matrix breaking these down by authority

Goal: feel how measures compute differently in different visual contexts.

Practice exercise 5 — Break it deliberately

This is the most useful exercise:

  1. Add a Removed Duplicates step on a column that's not unique — observe rows disappearing
  2. Change a relationship from one-to-many to many-to-many — observe count behaviours change
  3. Add a CALCULATE filter to a measure that conflicts with a slicer — observe the slicer being "overridden"
  4. Add a visual-level filter and a page-level filter on the same column with different values — observe what happens

Goal: see the exact mechanisms of distortion firsthand, so you recognise them in the wild.
